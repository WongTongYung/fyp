3.2 METHODOLOGY
         The development of the Automated Pickleball Point Counting System adopts the Waterfall

model. This approach organizes the project lifecycle into four different stages, which is the data
collection and preparation, core detection module, game logic and database development and
system integration and testing. This sequential approach ensures prioritization of the core Artificial
Intelligence (AI) component before the development of the application logic. This ensures that the
object detection model meets the necessary accuracy requirements before it is integrated into the
final software solution. Figure 3.2 shows the waterfall model for this project.

         The first phase focuses on a custom dataset. Data collection involves recording actual

gameplay using a standard USB webcam set to a 1080p resolution. As illustrated in Figure 3.3, the
camera is positioned at a fixed point at the rear center of the court, elevated approximately 1.5
meters high. This specific placement ensures a complete Field of View (FOV) of the court lines
and player movements while minimizing occlusion.

         This specific configuration was selected based on several key factors. The longitudinal
angle offers an optimal Field of View (FOV) that captures both the serving and receiving sides
within a single frame. Furthermore, positioning the camera at an elevated height minimizes the risk
of obstruction by the players compared to net-level or side-line configurations. Finally, the fixed
position of the camera creates a uniform tracking environment, ensuring the detection algorithm
can consistently monitor the ball across the entire court.

3.2.2 Phase 2: Core Detection Module (Yolov11 Training)
         In this phase, the training process focuses on maximizing the Mean Average Precision

(mAP). The model is fine-tuned on the custom dataset prepared in Phase 1 using the PyTorch deep
learning framework. mAP is a performance metric that calculates the average precision across
various recall levels, providing a single score that balances false positives and false negatives. This
metric is crucial for validating the model's consistency in detecting the small, high-velocity
pickleball across different frames. Specifically, the system aims to optimize mAP@50, and
hyperparameters will be iteratively adjusted to maximize this score, ensuring high-confidence
tracking in real-time.

3.2.3 Phase 3: Game Logic And Database Development
         A python language is used to construct a base game logic engine as a state machine. This

engine is responsible for enforcing the specific rules of pickleball, particularly the traditional side-
out scoring format, which requires tracking the serving teams score, the receiving teams score, and
the server number. Concurrently, a local storage solution is implemented using the SQLite database
engine. This database is designed to log game events accompanied by a precise timestamp. This
ensures the system works entirely offline and preserves match data for post-game analysis.
